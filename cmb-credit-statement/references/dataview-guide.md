# Dataview 动态计算使用指南

## 概述

账单报告使用 Obsidian Dataview 插件实现动态计算，修改交易记录后，所有统计数据会自动更新。

## 前置要求

### 1. 安装 Dataview 插件

1. 打开 Obsidian 设置
2. 进入「第三方插件」
3. 搜索「Dataview」
4. 点击安装并启用

### 2. 启用 JavaScript Queries

1. 打开 Dataview 插件设置
2. 找到「Enable JavaScript Queries」
3. 开启此选项

## 动态计算功能

### 自动更新的数据

报告中以下数据会根据交易明细自动计算：

#### 1. 账单概况
```
- 总消费: 自动汇总所有正数交易
- 总还款: 自动汇总所有负数交易
- 净消费: 总消费 + 总还款
- 额度使用率: 本期应还 / 信用额度 × 100%
```

#### 2. 分卡消费统计
```
按卡号自动统计:
- 交易笔数
- 消费金额（正数交易）
- 还款金额（负数交易）
- 净消费（消费 + 还款）
```

#### 3. 消费分类统计
```
按关键词自动分类:
- 餐饮: 餐、饭、食、咖啡、茶、美团、饿了么...
- 购物: 超市、商场、淘宝、京东、天猫...
- 交通: 滴滴、地铁、公交、加油...
- 娱乐: 电影、游戏、健身、旅游...
- 生活: 水电、话费、宽带、医疗...
- 其他: 未匹配的交易
```

## 使用场景

### 场景 1: 修正错误金额

**问题**: 发现某笔交易金额记录错误

**操作**:
1. 找到「交易明细」中的对应记录
2. 直接修改金额
3. 保存文件

**结果**: 
- 总消费自动更新
- 分卡统计自动更新
- 分类统计自动更新
- 额度使用率自动重算

**示例**:
```markdown
# 修改前
| 01/15 | 支付宝-超市 | ¥ 100.00 | 2693 |

# 修改后
| 01/15 | 支付宝-超市 | ¥ 150.00 | 2693 |
```

### 场景 2: 调整交易分类

**问题**: 某笔交易被错误分类

**操作**:
1. 修改交易描述，添加正确的分类关键词
2. 保存文件

**结果**: 
- 该交易自动归入正确分类
- 两个分类的统计都自动更新

**示例**:
```markdown
# 修改前（被分到"其他"）
| 01/20 | 某某公司 | ¥ 50.00 | 2693 |

# 修改后（归入"餐饮"）
| 01/20 | 某某公司-餐厅 | ¥ 50.00 | 2693 |
```

### 场景 3: 删除重复交易

**问题**: 发现重复记录的交易

**操作**:
1. 删除重复的交易行
2. 保存文件

**结果**: 
- 总消费自动减少
- 相关分类统计自动更新

### 场景 4: 补充遗漏交易

**问题**: 发现账单遗漏了某笔交易

**操作**:
1. 在对应分类下添加新的交易行
2. 按照表格格式填写: `| 日期 | 描述 | 金额 | 卡号 |`
3. 保存文件

**结果**: 
- 总消费自动增加
- 分卡统计自动更新
- 分类统计自动更新

**示例**:
```markdown
### 餐饮

| 日期 | 描述 | 金额 | 卡号 |
|------|------|------|------|
| 01/12 | 支付宝-餐厅A | ¥ 50.00 | 2693 |
| 01/15 | 支付宝-餐厅B | ¥ 80.00 | 2693 |
| 01/20 | 支付宝-餐厅C | ¥ 60.00 | 2693 |  ← 新增这行
```

## 表格格式要求

### 交易明细表格格式

```markdown
| 日期 | 描述 | 金额 | 卡号 |
|------|------|------|------|
| MM/DD | 交易描述 | ¥ 金额 | 4位卡号 |
```

**重要规则**:
1. 日期格式: `MM/DD` (例如: `01/15`)
2. 金额格式: `¥ 金额` (例如: `¥ 100.00`)
3. 金额必须保留两位小数
4. 还款用负数表示 (例如: `¥ -1,000.00`)
5. 卡号必须是4位数字

**正确示例**:
```markdown
| 01/15 | 支付宝-杭州某餐厅 | ¥ 50.00 | 2693 |
| 02/01 | 银联转账还款 | ¥ -1,000.00 | 1727 |
```

**错误示例**:
```markdown
| 1/15 | 餐厅 | 50 | 2693 |           ❌ 日期格式错误
| 01/15 | 餐厅 | ¥50.00 | 2693 |      ❌ 金额前缺少空格
| 01/15 | 餐厅 | ¥ 50 | 2693 |         ❌ 金额缺少小数
| 01/15 | 餐厅 | ¥ 50.00 | 269 |       ❌ 卡号不是4位
```

## 故障排查

### 问题 1: 统计数据不更新

**可能原因**:
- Dataview 插件未启用
- JavaScript Queries 未开启
- 表格格式不正确

**解决方法**:
1. 检查 Dataview 插件是否启用
2. 检查「Enable JavaScript Queries」是否开启
3. 检查交易记录格式是否符合要求
4. 尝试重新打开文件

### 问题 2: 显示 "Evaluation Error"

**可能原因**:
- JavaScript 代码语法错误
- 表格格式严重错误

**解决方法**:
1. 检查是否误删了 dataviewjs 代码块
2. 检查表格是否有格式错误
3. 尝试恢复到原始报告

### 问题 3: 某笔交易未被统计

**可能原因**:
- 表格格式不符合要求
- 金额格式错误
- 不在正确的表格中

**解决方法**:
1. 检查该行是否在「交易明细」章节下
2. 检查格式是否完全符合要求
3. 确保金额有 `¥` 符号和空格

## 最佳实践

### 1. 定期备份

在大量修改前，建议先备份原始报告。

### 2. 批量修改

如果需要修改多笔交易，建议：
1. 先在文本编辑器中准备好
2. 一次性粘贴到 Obsidian
3. 检查格式后保存

### 3. 验证统计

修改后，检查：
- 总消费是否合理
- 分卡统计是否正确
- 分类统计是否符合预期

### 4. 保持格式一致

- 统一使用两位小数
- 统一日期格式
- 统一描述风格

## 高级技巧

### 添加自定义计算

可以在报告末尾添加自定义 Dataview 查询：

```markdown
## 📊 自定义分析

```dataviewjs
// 计算平均单笔消费
const transactions = dv.current().file.lists
    .filter(l => l.text.match(/^\| \d{2}\/\d{2} \|/))
    .map(l => {
        const match = l.text.match(/¥\s*([-\d,]+\.\d{2})/);
        return match ? parseFloat(match[1].replace(',', '')) : 0;
    })
    .filter(t => t > 0);

const avgSpending = transactions.reduce((a, b) => a + b, 0) / transactions.length;

dv.paragraph(`平均单笔消费: ¥ ${avgSpending.toFixed(2)}`);
\```
```

## 总结

使用 Dataview 动态计算的优势：

✅ **实时更新**: 修改即生效，无需重新生成报告
✅ **灵活调整**: 可以自由添加、删除、修改交易
✅ **自动分类**: 修改描述后自动重新分类
✅ **准确可靠**: 基于实际数据计算，避免手动错误

记住：**修改交易明细，统计自动更新！**
